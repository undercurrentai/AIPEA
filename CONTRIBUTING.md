# Contributing to AIPEA

Thanks for your interest in AIPEA. This document describes how to contribute code, documentation, or security reports, and the review gates your contribution will pass through.

## Before You Start

1. **Read [`CLAUDE.md`](CLAUDE.md)** — it is the agent contract for this project, and captures the project's tier (Tier 2, Standard), token budget, and ASK-first triggers. Humans benefit from reading it too.
2. **Read [`SPECIFICATION.md`](SPECIFICATION.md)** — the canonical description of what AIPEA *is*.
3. **Check [`KNOWN_ISSUES.md`](KNOWN_ISSUES.md) and [`TODO.md`](TODO.md)** — your change may already be tracked, or may touch something that's in flight.

## Quality Gates

Run `make all` before pushing. It runs the same gates CI runs, plus auto-formatting.

| Gate | Command | Blocks merge? |
|---|---|---|
| Ruff format | `make fmt` (auto-fix) / `make lint` (check) | Yes |
| Ruff lint | `make lint` | Yes |
| mypy `--strict` on `src/aipea/` | `make type` | Yes |
| pytest + coverage floor 75% | `make test` | Yes |
| Security rules (Bandit-aligned) | `make sec` | No, but should be clean |

PRs failing any blocking gate will not be merged. If your PR legitimately needs to bypass a gate, include an approved exception memo referencing the rationale.

## Change Classes

Every PR declares a change class in its description:

| Class | Description | Review depth |
|---|---|---|
| **A** (style) | Formatting, typos, docstrings, comment cleanup — no behavioral change | Light |
| **B** (refactor) | Internal refactors that preserve observable behavior | Normal |
| **C** (behavioral) | Any change to observable behavior, public API, or security/compliance paths | Full review + regression tests required |

Class **C** changes touching `src/aipea/security.py`, `src/aipea/__init__.py`, `pyproject.toml`, or `.github/workflows/**` additionally pass through the **Second-Reviewer Gate** (see below).

## Second-Reviewer Gate (Security-Critical Paths)

AIPEA currently has a single human maintainer. To mitigate bus-factor risk on security-critical code, PRs touching the following paths must pass an **automated triple-AI review gate** as a required CI status check before merge:

- `src/aipea/security.py`
- `src/aipea/__init__.py` (public API surface)
- `pyproject.toml` (dependency changes)
- `.github/workflows/**` (CI / release pipeline)

The gate is implemented as `.github/workflows/ai-second-review.yml` and runs three parallel jobs on every PR that touches a gated path:

1. **`gpt-review`** — calls GPT 5.4 Pro via the OpenAI API with the PR diff and posts findings as a PR comment.
2. **`codex-review`** — runs Codex CLI in non-interactive `exec` mode against the diff and posts a second PR comment.
3. **`claude-review`** — runs Claude Opus 4.6 via `anthropics/claude-code-action@v1`, reading the project's `CLAUDE.md` natively for full context, and posts a third PR comment.

All three jobs are required status checks on the `main` branch via branch protection. Merge is blocked until all jobs report success. `@joshuakirby` remains the accountable human reviewer via `.github/CODEOWNERS`.

**Verdict enforcement.** Each reviewer posts a structured comment with a `Verdict` of `APPROVE`, `REQUEST_CHANGES`, or `COMMENT`. The workflow enforces this programmatically: a `REQUEST_CHANGES` verdict fails the CI job, blocking merge via branch protection. `APPROVE` and `COMMENT` both pass. To unblock a PR after `REQUEST_CHANGES`, address the blocking concerns, push a fix, and the reviewer will re-run automatically on the new commit.

**Fail-closed policy.** If any AI reviewer is unavailable (API outage, quota exhaustion, transient 5xx), its CI job will fail. The PR author may document the unavailability in a PR comment and request an override via `@joshuakirby`, who is the CODEOWNERS approver. Do **not** `--admin`-merge a security-gated PR without posting that justification in the PR first.

**Why this gate exists.** The 2026-04-11 investor review sequence identified bus-factor-1 as the single largest engineering risk in AIPEA. The automated triple-AI gate is the interim answer until a contracted human second reviewer is available. See `docs/ROADMAP.md` §P5a for the full rationale.

## AI-Assisted Contributions

If your contribution was authored or materially assisted by an AI coding tool (Claude Code, Copilot, Cursor, Codex, etc.), disclose it.

- Include the tool name and version in the PR description under a `## AI-Assisted Development` section.
- For Class C changes, additionally describe what the human contributor verified (test coverage, reasoning, edge cases) and what the AI generated verbatim.
- AI-authored commits should use the `Co-Authored-By:` trailer with a non-PII noreply address, e.g. `Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>`.
- Claims generated by AI ("this is faster", "this is safer", "this fixes X") must be independently verified by the human contributor before being added to PR text or commit messages.

This policy exists because the AEGIS governance framework ([aegis-governance repo](https://github.com/undercurrentai/aegis-governance)) — which AIPEA pairs with — tracks AI-assisted development as an engineering-standards signal. Your PR becomes part of that tracking.

## Reporting Security Vulnerabilities

**Do NOT open a public GitHub issue for security vulnerabilities.** See [`SECURITY.md`](SECURITY.md) for the disclosure policy, including:

- GitHub Security Advisory flow: <https://github.com/undercurrentai/AIPEA/security/advisories/new>
- Email: `security@undercurrentholdings.com`
- Response SLAs (critical 7 days / high 14 days / medium 30 days)
- Honest scope statement of what AIPEA's compliance modes do and do not enforce

## Dependency Additions

AIPEA maintains a deliberately minimal dependency surface. Core modules (`security`, `knowledge`, `config`, `_types`, `quality`) are **stdlib-only**. `search` uses stdlib + `httpx`. The optional CLI adds `typer` and `rich`.

Any new runtime or dev dependency requires an ASK-first approval per `CLAUDE.md §3.3`. Open an issue first proposing the dependency, its license, its transitive dependencies, and why an stdlib or existing-dep alternative won't work. GPL/LGPL dependencies are refused (AIPEA is MIT-licensed).

## Service or Tooling Additions

Any new service or long-running tool shipped as part of AIPEA must include:

- OpenAPI/Proto spec (for HTTP/RPC surfaces)
- Performance SLOs
- Observability fields in structured logs (`request_id`, `seed_id`, `dataset_hash` where applicable)

AIPEA is a library today; this gate applies if/when we ship the service-mode wrapper in the v1.5.0+ timeframe.

## Deviations from Standard Tooling

Any deviation from `ruff` / `mypy` / `pytest` / `hatchling` requires an ADR in [`docs/adr/`](docs/adr/) documenting the rationale and escape hatch.

## Branch and Commit Conventions

- **Branch naming**: `<type>/<short-description>` — e.g., `feat/adaptive-learning`, `fix/regex-redos`, `docs/security-policy`
- **Commit messages**: [Conventional Commits](https://www.conventionalcommits.org/) — `feat:`, `fix:`, `test:`, `chore:`, `docs:`, `release:`
- **Scope**: Use `(module)` when touching a single module — e.g., `fix(security): handle empty patterns`

## Release Process

See `CLAUDE.md §6.5` for the full release playbook. TL;DR:

1. `make ci` green locally
2. Bump version in `pyproject.toml` + `src/aipea/__init__.py`
3. Cut `CHANGELOG.md` `[Unreleased]` → `[X.Y.Z] - DATE`
4. Commit, PR, CI, merge
5. `gh release create vX.Y.Z` — `publish.yml` auto-publishes to PyPI via OIDC Trusted Publisher

PyPI releases are ASK-first per `CLAUDE.md §2.2`. Do not cut a release without maintainer approval.
